## For adapter consumers

You are wiring a governance layer into an autogen agent. Here is what you need to know.

### What this adapter is

A single async hook — `governedToolCall()` — that autogen consumers register as `before_tool_call`. It enforces three checks in order: identity, authorization, optional trust. First failure returns a `Deny` with the failed check name and evidence. All pass returns an `Allow` with normalized context.

### What this adapter is not

- Not an identity issuer. Bring your own passport from APS SDK, AgentID, did:key, SPIFFE, or anything else that produces Ed25519-signed artifacts.
- Not a policy engine. The scope-match is a leaf-hop string-and-token comparison. For richer policy, compose with SINT or your own engine on top of the `Allow` result.
- Not an autogen plugin. It is a plain async function. Wire it into whatever hook surface your autogen version exposes.

### Wiring into autogen

```python
from autogen_governance_adapter import governedToolCall

async def before_tool_call(tool_name, tool_args, *, passport, delegation_chain):
    decision = await governedToolCall(
        passport=passport,
        delegation_chain=delegation_chain,
        tool_name=tool_name,
        tool_args=tool_args,
        principal_jwks=my_principal_jwks,
        trust_provider=my_trust_provider,
        min_trust_score=0.7,
    )
    if decision.kind == "deny":
        raise PermissionError(f"{decision.failed_check}: {decision.reason}")
```

The `passport` and `delegation_chain` arguments are dataclasses defined in `passport.py` and `delegation.py`. Any identity layer that can produce those shapes works.

### Scope-bound substrate

Inference-substrate requirements ride inside delegation scope. Scope tokens:

- `action:*`, `action:read`, `action:call`
- `resource:<name>`
- `inference_substrate:<approved_set_id>`
- `min_capability_tier:<T>`

The hook checks scope-match. A substrate-aware trust provider can treat the substrate token as a query key. No parallel capability-tier gate.

### Trust providers

Implement the two-method `TrustProvider` Protocol from `trust_provider.py`. Ship as a separate package. Open a PR against `providers/<name>/` with a README + pointer to your package. The adapter stays provider-agnostic.

## For coding agents working on this repo

You are modifying this repo. Here is the contract.

### Dev environment

```bash
python3.12 -m venv .venv
.venv/bin/pip install -e ".[dev]"
.venv/bin/pytest
.venv/bin/black --check src/ tests/
.venv/bin/ruff check src/ tests/
```

All four commands must exit 0 before any push. CI runs them across Python 3.10, 3.11, 3.12.

### What to change

- Bug fix: edit the code, add a regression test, run all four commands, commit.
- New primitive: add it to the right module (identity → `passport.py`, authorization → `delegation.py`, trust → `trust_provider.py`, integration → `hook.py`), export from `__init__.py`, add tests, document in README if it is part of the public surface.
- New trust provider: **do not add it in this repo**. Open a PR against `providers/<name>/` with a README and a dependency link. The adapter itself stays provider-agnostic.

### What not to do

- Do not import `agent-passport-system` or any SDK as a hard dep. This adapter is standalone.
- Do not add MoltBridge-specific or MolTrust-specific code to the core. Trust providers live behind the Protocol.
- Do not add a capability-tier check as a separate gate. Substrate and tier are scope tokens.
- Do not publish to PyPI. That is a human-driven step with Touch ID.
- Do not scaffold providers on spec. One empty placeholder for MoltBridge. Other providers land as their own PRs from their maintainers.
- Do not announce work on GitHub threads. Report back locally and let the human handle the thread.

### PR rules

- One logical change per PR. Keep diffs reviewable.
- No mass reformatting in a feature PR. Formatting PRs are separate.
- Tests for every behavior change. Fixtures are deterministic (seeded keys) — no flakes allowed.
- Verify, do not claim. Run the four commands, paste output in the PR body.

### Test fixtures

Fixtures under `tests/fixtures/` are regenerated by `scripts/generate_fixtures.py` using fixed seeds. If you add a new scenario, regenerate, commit the JSON, and add the generator recipe to that script.
